### DEFECT-1 🔴 高风险 — AssetType/AssetStatus.fromCode 无 orElse，未知编码直接崩溃

**文件**: `lib/domain/entities/asset_enums.dart:22, 36`
**函数**: `AssetType.fromCode`, `AssetStatus.fromCode`
**类型**: StateError / 未捕获异常

**风险代码**:
```dart
// line 22
AssetType.values.firstWhere((e) => e.code == code);
// line 36
AssetStatus.values.firstWhere((e) => e.code == code);
```

**根因**: `firstWhere` 在无匹配时直接抛出 `StateError`，无 `orElse` 回退。调用点在 `asset_mapper.dart:18,26`，处于 `DriftAssetRepository.watchAll()` 等 Stream 的 `.map()` 回调中，Stream 没有 catch 保护。若数据库存在历史遗留数据、手动 SQL 写入或版本升级后的新枚举值，反序列化时 Stream 立即关闭并传播错误，导致资产列表、仪表盘等所有依赖模块崩溃。

**攻击路径**: DB 存储未知 assetType/status → mapper.toDomain 调用 fromCode → StateError → Stream 关闭并传播 error → Riverpod provider 进入 error 状态 → 所有资产相关 UI 崩溃

**修复建议**:
```dart
static AssetType fromCode(String code) =>
  values.firstWhere(
    (e) => e.code == code,
    orElse: () => throw ValidationError('unknown asset type: $code'),
  );

// 或更安全的返回 null 策略：
static AssetType? tryFromCode(String code) =>
  values.where((e) => e.code == code).firstOrNull;
```

### DEFECT-2 🔴 高风险 — database.dart file.length() catchError→0 可能误删真实数据库

**文件**: `lib/data/db/database.dart:470`
**函数**: `_openConnection`
**类型**: 异常吞没 / 数据丢失

**风险代码**:
```dart
final fileSize = await file.length().catchError((_) => 0);
if (fileSize < 4096) {
  await file.delete();  // 这行可能误删真实 DB！
```

**根因**: `file.length()` 在 I/O 错误（权限问题、文件系统异常、文件锁定等）时抛出异常，`catchError` 将其静默替换为 `0`。`fileSize = 0 < 4096` 触发删除逻辑，导致可能包含真实用户数据的数据库文件被删除。设计意图是仅删除「无效的空文件」，但 I/O 错误场景下该判断失效，造成不可恢复的数据丢失。

**修复建议**:
```dart
// 方案1：让 I/O 错误自然传播
final fileSize = await file.length(); // 移除 catchError

// 方案2：I/O 错误时保守跳过删除
int fileSize;
try {
  fileSize = await file.length();
} catch (e) {
  // 无法获取文件大小时，保守地不删除
  debugPrint('_openConnection: cannot get fileSize: $e');
  fileSize = 4096; // 假设有效，跳过删除
}
```

### DEFECT-4 🔴 高风险 — database.dart onUpgrade 多步迁移非原子，崩溃导致 schema 不一致

**文件**: `lib/data/db/database.dart:117`
**函数**: `onUpgrade`
**类型**: 数据不一致 / 迁移失败

**风险代码**:
```dart
onUpgrade: (m, from, to) async {
  if (from < 2) { await m.addColumn(...); }
  if (from < 8) {
    await m.createTable(...);
    await m.customStatement('INSERT INTO ...');
    await m.customStatement('DELETE FROM ...');
  }
  // 16 个版本步骤串行执行，无事务包裹
}
```

**根因**: Drift 的 `MigrationStrategy.onUpgrade` 不自动包装事务。16 个迁移版本中多个步骤包含多条 DDL+DML 语句（如 v8 的建表+迁移数据+删旧表，v14 的重命名+建新表+数据迁移+删旧表），若应用在中间某步崩溃，SQLite `user_version pragma` 不会在迁移完成后才更新（视 Drift 实现而定），或已执行的部分 DDL 无法回滚，导致 schema 永久处于中间态，下次启动时迁移被跳过，应用持续崩溃。

**修复建议**:
```dart
onUpgrade: (m, from, to) async {
  await m.database.transaction(() async {
    if (from < 2) { ... }
    // ... 所有迁移步骤
  });
},
```

**风险代码**:
```dart
final snap = (jsonDecode(plain.valueOrNull!) as Map)
    .cast<String, dynamic>()
```

**根因**: `v as List` 在数据结构非预期时抛出 `TypeError`，被外层 `catch (e)` 包装为 `StorageError('import failed: $e')`，用户无法区分「密码错误」与「备份数据结构损坏」。虽然 AES-GCM MAC 校验理论上阻止外部篡改，但自产合法备份在格式版本变更或写入不完整时也可触发此路径。

**修复建议**:
```dart

if (rawMap == null) {
  return const Err(ValidationError('backup data corrupted: root not a map'));
}
final snap = rawMap.cast<String, dynamic>().map((k, v) {
  if (v is! List) {
    throw ValidationError('backup data corrupted: table $k is not a list');
  }
  return MapEntry(k, (v).cast<Map<String, dynamic>>());
});
```
