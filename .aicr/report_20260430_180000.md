## 缺陷详情

### DEFECT-1 🔴 高风险 — 数据库文件误删（marker 机制缺陷）

**位置**：`lib/data/db/database.dart`（约第 463 行）
**函数**：`_openConnection` (LazyDatabase callback)
**风险代码**：`if (await file.exists() && !await marker.exists()) { await file.delete();`

**问题描述**：
应用首次使用加密数据库时，会在同目录写入一个 `gwp.db.encrypted` marker 文件，用于区分"已迁移到加密版本"的状态。若 marker 文件因以下任意原因丢失：
- 文件系统异常（Android 分区 OOM、存储清理工具误删）
- 用户从备份恢复时仅恢复了数据库文件，marker 未被包含
- 操作系统 Bug 导致文件目录部分丢失

则下次应用启动时，该判断条件（"DB 文件存在 && marker 不存在"）被意外触发，**无条件删除整个加密数据库**，造成不可恢复的全量数据丢失。代码无任何二次确认、备份或校验机制。

**修复建议**：
1. 在 marker 文件中存储版本+时间戳（如 `{"v":1,"created":"2024-..."}`），在删除前验证 DB 文件的加密头魔数（SQLCipher 加密数据库头部有固定特征）；
2. 或将删除逻辑改为：仅在 **首次安装**（即应用数据目录为空）时执行，而不是依赖 marker 文件存在性判断；
3. 至少在执行删除前备份到临时路径，以便用户通过支持入口恢复。

### DEFECT-2 🔴 高风险 — TransferAssetUseCase 生产端未注入事务

**位置**：`lib/domain/usecases/transfer_asset.dart`（约第 68 行）
**函数**：`_defaultTx`
**风险代码**：`static Future<T> _defaultTx<T>(Future<T> Function() fn) => fn();`

**问题描述**：
`TransferAssetUseCase` 接受可注入的 `transaction` 参数以实现原子转账（创建目标资产 → 软删除源资产 → 写两条成本历史，共 4 步独立 SQL）。然而生产端注册代码（`lib/features/asset/presentation/asset_providers.dart` 约第 132-141 行）构造该 UseCase 时**未传入** `transaction:` 参数，导致 `_tx` 绑定到 `_defaultTx`（直接执行 `fn()`，无任何事务包裹）。

任意中间步骤失败时：
- 步骤 4 成功、步骤 5 失败 → 目标资产已创建，源资产未删除，用户资产虚增
- 步骤 5 成功、步骤 6/7 失败 → 资产迁移完成但成本历史不一致

**修复建议**：
在 `transferAssetUseCaseProvider` 的构造调用中，注入数据库事务函数：
```dart
TransferAssetUseCase(
  ...,
  transaction: <T>(fn) => ref.read(appDatabaseProvider).transaction(fn),
)
```

### DEFECT-3 🟡 中风险 — listRecent / listForTrend 无异常处理

**位置**：`lib/data/repositories/drift_asset_price_history_repository.dart`（约第 25、31 行）
**函数**：`listRecent`、`listForTrend`
**风险代码**：`final rows = await _dao.listRecent(limit: limit);`

**问题描述**：
`listRecent` 和 `listForTrend` 方法直接调用 DAO，无 try/catch 包裹，与同文件的 `record()` 方法（有 try/catch + Result 返回）形成不一致的错误处理策略。若 DAO 层抛出异常（数据库加密密钥错误、磁盘满、SQLite corrupt），异常将直接向上冒泡。调用方为 `dashboard_providers.dart` 的两处 Riverpod FutureProvider（`netWorthTrendProvider`、`accountNetWorthTrendProvider`），Provider 会进入 AsyncError 状态，若 UI 层未对 AsyncError 做防御处理，趋势图页面将出现白屏或错误界面。

**修复建议**：
为 `listRecent` 和 `listForTrend` 添加 try/catch，返回空列表降级（或改为 `Result<List<...>, AppError>` 类型与 `record` 保持一致）：
```dart
Future<List<AssetPriceHistoryPoint>> listRecent({required int limit}) async {
  try {
    final rows = await _dao.listRecent(limit: limit);
    return rows.map(mapper.fromRow).toList();
  } catch (e) {
    return const [];
  }
}
```

### DEFECT-4 🟡 中风险 — querySeriesForPair 静默吞掉存储错误

**位置**：`lib/data/repositories/drift_exchange_rate_repository.dart`（约第 42 行）
**函数**：`querySeriesForPair`
**风险代码**：`return const [];`（catch 块中）

**问题描述**：
`querySeriesForPair` 的 catch 块仅在 `kDebugMode` 下输出日志，然后返回空列表。代码自身注释已承认此为已知问题（"DB 故障不应被静默吞掉"）。唯一调用方 `check_rate_alerts.dart` 虽然对 `querySeriesForPair` 自身加了 try/catch，但 repository 内部已吞掉异常、永远不抛出，外层 catch 永远不触发。结果：DB 故障时汇率预警链路静默失效，用户应收到的阈值告警被丢弃；release 构建下日志被关闭，问题完全不可见。

**修复建议**：
将返回类型改为 `Result<List<ExchangeRate>, AppError>` 以区分"无数据"与"查询失败"；或至少在 release 模式下也通过 `log` package 记录错误，并修改注释中的 TODO 为实际 issue 追踪。

### DEFECT-5 🟡 中风险 — ValuateAssetUseCase 资产更新与审计日志非原子

**位置**：`lib/domain/usecases/valuate_asset.dart`（约第 77-110 行）
**函数**：`ValuateAssetUseCase.call`
**风险代码**：`await _assets.update(updated)` ... `await _priceHistory.record(point)` (两个独立调用)

**问题描述**：
资产价格持久化（`_assets.update`）与价格历史审计日志写入（`_priceHistory.record`）是两次独立的数据库操作，之间无事务包裹。若 `_assets.update` 成功后 `_priceHistory.record` 失败（约束冲突、磁盘错误等），方法返回 `Err`，但资产的 `currentPrice`/`marketValue`/`valuationTime` 已被永久修改，而 `asset_price_history` 表中无对应记录，形成数据不一致：价格已更新但无审计轨迹。调用方收到错误后无回滚措施。

**修复建议**：
将两步操作包裹在同一数据库事务中，或先写审计日志再更新资产（失败时整体回滚）：
```dart
await _db.transaction(() async {
  await _assets.update(updated);
  await _priceHistory.record(point);
});
```

### DEFECT-6 🟡 中风险 — setPin 中 salt+hash 两次写入非原子

**位置**：`lib/core/auth/pin_store.dart`（约第 103-104 行）
**函数**：`setPin`
**风险代码**：`await _storage.write(_kSalt, ...); await _storage.write(_kHash, ...);`

**问题描述**：
`setPin` 分两次独立写入 `_kSalt` 和 `_kHash`，`FlutterSecureStorage` 无批量写入或事务 API，两次调用之间不具备原子性。若进程在第一次写入（salt）成功、第二次写入（hash）之前崩溃，则：
- `hasPin()` 检查两个键都存在才返回 true，此时因 hash 为 null 返回 false
- 用户的 PIN 记录等效被抹除，下次启动被当作首次设置
- 实际上 salt 仍存在，如果稍后 setPin 使用新 salt，旧 salt 已无用途

后果：用户在更换 PIN 过程中恰好遇到进程崩溃（如 Android OOM Kill）会丢失 PIN，强制重置，虽低概率但后果不可逆。

**修复建议**：
将 salt 和 hash 序列化为单个 JSON 字符串后一次性写入：
```dart
final payload = jsonEncode({'salt': base64Encode(salt), 'hash': base64Encode(bytes)});
await _storage.write(_kPin, payload);
```
同步更新 `hasPin()`、`verifyPin()` 的读取逻辑。
