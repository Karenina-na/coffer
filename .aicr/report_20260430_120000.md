## 缺陷详情

### 缺陷 1 — 计时旁道（pin_store.dart:130）

**严重程度**：🟡 中风险
**函数**：`PinStore.verifyPin`

**问题代码**：
```dart
// lib/core/auth/pin_store.dart:128-131
final actual = await derived.when(
  ok: (k) async => await k.extractBytes(),
  err: (e) async => <int>[],   // ← 问题所在
);
```

**原因分析**：
当 `_kdf.derive()` 失败（err 分支）时，`actual` 被赋为空列表。`_constantTimeEquals(expected, actual)` 因 `a.length != b.length` 直接返回 `false`，跳过逐字节比对循环。
正常路径：需等待 Argon2id 完成（~几百ms）再做常量时间比对；
错误路径：直接比较长度后返回 false（几乎无延迟）。
这使攻击者能够通过计时区分两种路径，违反常量时间比对的设计意图。

**现有保护**：
- `maxFailBeforeLock = 5` 次限制 + 30s 冷却时间，大幅限制利用窗口
- KDF 失败属于极罕见运行时错误

**修复建议**：
```dart
err: (e) async {
  // 生成与 expected 等长的随机字节做虚假比对，保持时序一致
  final dummy = List<int>.filled(expected.length, 0);
  _constantTimeEquals(expected, dummy); // 消耗等量时间
  return <int>[];
},
```
或直接在 err 分支返回 `Err(CryptoError('KDF failed'))`，让调用方区分 KDF 错误与 PIN 错误。

### 缺陷 2 — OOM 防护绕过（backup_restore.dart:127）

**严重程度**：🔴 高风险
**函数**：`ImportBackupUseCase.call`

**问题代码**：
```dart
// lib/domain/usecases/backup_restore.dart:127
if (package.length > backupMaxBytes) {
  return const Err(ValidationError('backup too large'));
}
```

**原因分析**：
`String.length` 在 Dart 中返回 UTF-16 代码单元数。对于包含大量 3 字节 UTF-8 字符（如汉字）的字符串：

- 一个 CJK 汉字：`package.length += 1`，但 `utf8.encode(s).length += 3`
- 攻击者可构造 `package.length = 50MB` 但 `UTF-8 字节数 ≈ 150MB` 的字符串
- 绕过限制后 `jsonDecode` 尝试解析超大字符串，触发 OOM crash

**修复建议**：
```dart
// 在读取文件时限制字节数，而非加载为 String 后再检查
if (utf8.encode(package).length > backupMaxBytes) {
  return const Err(ValidationError('backup too large'));
}
// 或更好：在 File.readAsBytes() 阶段限制
```

### 缺陷 3 — 备份恢复中 fromJson 异常无保护（db_snapshot.dart:149）

**严重程度**：🔴 高风险
**函数**：`_batchInsert` / `restore`

**问题代码**：
```dart
// lib/data/backup/db_snapshot.dart:145-152
await _db.batch((b) {
  for (final r in rows) {
    b.insert(table, fromJson(r));  // ← fromJson 无 try/catch
  }
});
```

**原因分析**：
`fromJson(r)` 是 Drift 生成的静态方法，内部做强制类型转换。若备份 JSON 中任意行数据存在类型不符：
- `as int` 实际为 `null` → `TypeError: Null is not a subtype of int`
- 必填字段缺失 → `CastError`

该异常在 `batch` 同步 callback 中抛出，穿透至 `transaction()` 的 Future，Drift 执行 ROLLBACK（好事），但 `restore()` 返回的 Future 以异常（而非 `Err`）完成，调用方无法收到结构化错误，备份文件中只要有一行数据损坏就导致整个恢复失败。

**修复建议**：
```dart
Future<void> _batchInsert<T extends DataClass>(...) async {
  await _db.batch((b) {
    for (final r in rows) {
      try {
        b.insert(table, fromJson(r));
      } catch (e) {
        throw StorageError('Failed to deserialize row in $table: $e');
      }
    }
  });
}
```
或在 `restore()` 调用 `_batchInsert` 时包裹 `try-catch(e)` 并转为 `Err(StorageError(...))` 返回。

### 缺陷 4 — 历史价格逐点写入无原子性（refresh_asset_price.dart:286）

**严重程度**：🟡 中风险
**函数**：`RefreshAssetPriceUseCase.refreshHistory`

**问题代码**：
```dart
// lib/domain/usecases/refresh_asset_price.dart:282-288
for (final point in series.points) {
  final rec = await _priceHistory.record(point);
  if (rec.isErr) return Err(rec.errorOrNull!);  // ← 部分写入后直接返回
}
```

**原因分析**：
若 30 天历史数据中第 15 天写入失败：
1. 前 14 天数据已写入数据库（不可回滚）
2. 函数返回 `Err`，`refreshAll` 将该资产记入 `failed`
3. 后续重刷时，`record()` 使用 `sourceKey = assetId:dayKey:source` 做 `INSERT OR IGNORE`，已写的 14 天会被跳过
4. 缺失的第 15~30 天在下次成功前持续产生空洞

**修复建议**：
```dart
// 方案一：包裹在数据库事务中
await _db.transaction(() async {
  for (final point in series.points) {
    final rec = await _priceHistory.record(point);
    if (rec.isErr) throw rec.errorOrNull!;
  }
});

// 方案二：采用 INSERT OR REPLACE 语义，失败时继续（容忍部分点失败）
int successCount = 0;
for (final point in series.points) {
  final rec = await _priceHistory.record(point);
  if (rec.isOk) successCount++;
}
if (successCount == 0) return Err(...);
```
